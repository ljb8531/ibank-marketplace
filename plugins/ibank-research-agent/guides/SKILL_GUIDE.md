## 스킬 분리 판단 기준

### 스킬을 만들어야 하는 3가지 경우

**경우 1 — 반복되는 도구 사용 패턴에 가드레일이 필요할 때**

에이전트가 도구(WebSearch, mcp_web_fetch, browser)를 호출하는데, 매번 동일한 절차와 판단 기준을 따라야 하는 경우. 이걸 에이전트 프롬프트에만 적어두면 10번 호출 중 3번은 절차를 빼먹거나 판단 기준을 무시합니다. 스킬로 분리하면 "이 절차를 반드시 따라라"를 강제할 수 있습니다.

**경우 2 — 프롬프트가 너무 길어서 에이전트의 본질적 판단력이 흐려질 때**

에이전트 프롬프트에 "너의 역할은 이거고, 판단 기준은 이거고, 출력 형식은 이거고, 도구 사용법은 이거고..."를 전부 넣으면, 프롬프트가 길어질수록 에이전트가 핵심 지시를 놓치는 확률이 올라갑니다. 절차적이고 반복적인 부분을 스킬로 빼면, 에이전트 프롬프트는 "판단과 오케스트레이션"에 집중할 수 있습니다.

**경우 3 — 다른 에이전트나 다른 플로우에서도 재사용될 때**

같은 작업을 여러 에이전트가 수행하거나, Flow 1뿐 아니라 Flow 2, Flow 3에서도 동일한 작업이 필요한 경우. 스킬로 분리해두면 복사/붙여넣기 없이 재사용할 수 있습니다.

### 스킬을 만들지 말아야 하는 경우

해당 에이전트에서 한두 번만 실행되는 단순 작업, LLM의 자연어 추론 능력으로 충분히 처리되는 분석/판단 작업, 프롬프트 수준에서 지시하면 잘 따르는 출력 형식 규정 — 이런 것들은 스킬로 분리하면 오히려 호출 오버헤드만 늘어납니다.

## 에이전트별 스킬 분리 분석

### Service Scout — 스킬 분리 필요

Service Scout은 4개 에이전트 중 유일하게 도구를 사용하는 에이전트이고, 도구 사용 절차가 복잡하며 반복적입니다. 서비스 하나당 동일한 수집 절차를 10~15번 반복해야 합니다. 여기서 가드레일이 없으면 수집 품질이 들쭉날쭉해집니다.

| 스킬 후보 | 분리 필요? | 이유 |
|---|---|---|
| **서비스 탐색 (검색)** | **Yes** | 검색 전략(5가지 패턴), 비교 글 2차 추출, must_include 검증 등 절차가 복잡하고 매번 동일하게 수행해야 함. 가드레일 필요. |
| **페이지 수집** | **Yes** | mcp_web_fetch → 부족하면 browser 전환 판단, 500자 기준, 수집 실패 처리 등 반복 절차에 가드레일 필요. 또한 Feature Analyzer/UXUI Analyzer가 추가 수집 요청 시에도 동일 절차 재사용. |
| **네비게이션 구조 수집** | **Yes** | browser를 사용한 메뉴 펼치기, DOM 스냅샷에서 GNB 추출 등 특수한 절차. 일반 페이지 수집과 다른 패턴. |
| 서비스 프로필 정리 | No | LLM의 텍스트 정리 능력으로 충분. 출력 형식만 프롬프트에 명시하면 됨. |

### Feature Analyzer — 스킬 분리 제한적

Feature Analyzer는 도구를 사용하지 않고 분석만 수행합니다. 대부분의 작업이 LLM의 추론 능력으로 처리됩니다. 다만 하나의 작업에서 가드레일이 필요합니다.

| 스킬 후보 | 분리 필요? | 이유 |
|---|---|---|
| 기능 추출 | No | LLM이 텍스트에서 기능을 추출하는 것은 기본 능력. 프롬프트 지시로 충분. |
| 기능 분류 | No | 카테고리 체계가 프롬프트에 명시되어 있으면 충분. |
| **기능 비교 매트릭스 생성** | **Yes** | 10~15개 서비스 × 수십 개 기능의 매트릭스를 일관된 기준(●/○/△/✕)으로 만들어야 함. 서비스가 많아지면 에이전트가 앞쪽 서비스와 뒤쪽 서비스에 다른 기준을 적용하는 일이 발생함. 매트릭스 작성 규칙을 스킬로 분리하면 일관성 보장. |
| 시사점 도출 | No | LLM의 분석 능력이 핵심인 작업. 스킬로 분리하면 오히려 유연성이 떨어짐. |
| 데이터 부족 판단 | No | 프롬프트에 "부족하면 명시하라"고 적으면 충분. |

### UXUI Analyzer — 스킬 분리 제한적

UXUI Analyzer도 도구를 사용하지 않습니다. 분석 관점이 5가지(정보 구조, 사용자 동선, 콘텐츠 전략, 인터랙션 패턴, 톤앤매너)로 나뉘는데, 이것을 각각 스킬로 만들면 과도합니다. 다만 특수한 부분이 하나 있습니다.

| 스킬 후보 | 분리 필요? | 이유 |
|---|---|---|
| 정보 구조 분석 | No | 네비게이션 데이터를 읽고 분석하는 것은 LLM 기본 능력. |
| 사용자 동선 분석 | No | 메인 페이지 콘텐츠 순서를 분석하는 것은 프롬프트 지시로 충분. |
| 콘텐츠 전략 분석 | No | 텍스트 분석 능력으로 처리. |
| 인터랙션 패턴 분석 | No | DOM 스냅샷에서 패턴을 추출하는 것은 프롬프트 지시로 충분. |
| 톤앤매너 분석 | No | 텍스트 어조 분석은 LLM의 강점. |
| **UXUI 비교표 생성** | **Yes** | Feature Analyzer의 매트릭스와 동일한 이유. 서비스 간 UXUI 비교를 일관된 기준으로 수행해야 함. |

### Report Writer — 스킬 분리 불필요

Report Writer는 전달받은 분석 결과를 종합 정리하는 역할입니다. 모든 작업이 LLM의 문서 작성 능력으로 처리되며, 반복적 도구 사용도 없습니다. 리포트 구조와 작성 원칙은 프롬프트에 명시하면 충분합니다.

| 스킬 후보 | 분리 필요? | 이유 |
|---|---|---|
| 리포트 작성 | No | 프롬프트에 구조와 원칙을 명시하면 충분. 스킬로 분리하면 유연성만 떨어짐. |
| 교차 분석 | No | 두 분석 결과를 엮는 것은 LLM 추론의 핵심 가치. 절차화하면 안 됨. |

## 만들어야 할 스킬 목록

결론적으로 5개 스킬입니다.

| 스킬명 | 소속 에이전트 | 분리 이유 | 재사용 가능성 |
|---|---|---|---|
| **search-services** | Service Scout | 검색 전략 5패턴 + 비교 글 2차 추출 + must_include 검증. 절차 복잡, 가드레일 필수. | Flow 1 내에서 반복 호출. Flow 2 이후에서도 추가 서비스 탐색 시 재사용. |
| **crawl-page** | Service Scout | mcp_web_fetch → browser 전환 판단, 500자 기준, 실패 처리. 서비스당 3~4회 반복. | 모든 Flow에서 페이지 수집이 필요할 때 재사용. 가장 범용적 스킬. |
| **scan-navigation** | Service Scout | browser 메뉴 펼치기, GNB 구조 추출. 일반 수집과 다른 특수 패턴. | UXUI 분석에서 추가 네비게이션 탐색 시 재사용 가능. |
| **build-feature-matrix** | Feature Analyzer | 대량 서비스 × 기능의 일관된 매트릭스 생성. 평가 기준(●/○/△/✕) 일관성 보장. | 기능 비교가 필요한 모든 분석에서 재사용. |
| **build-uxui-comparison** | UXUI Analyzer | 서비스 간 UXUI 비교를 일관된 기준으로 수행. 비교 항목과 평가 기준 표준화. | UXUI 비교가 필요한 모든 분석에서 재사용. |

## 각 스킬 상세 설계

### 스킬 1 — search-services

```
목적: 검색 키워드와 전략에 따라 웹 검색을 수행하고,
      벤치마킹 대상 서비스 목록을 추출하여 반환한다.

소속: Service Scout

사용 도구: WebSearch, mcp_web_fetch (비교 글 수집 시)

가드레일:
  1. 반드시 5가지 검색 패턴을 모두 수행한다 (하나라도 빼먹지 않는다)
     - 직접 검색
     - 비교/리뷰 검색
     - 특화 키워드 검색
     - scope에 따른 언어별 검색
     - must_include 서비스 직접 검색 (결과에 없을 경우)
  2. 비교 글이 발견되면 반드시 mcp_web_fetch로 본문을 수집하여
     추가 서비스를 추출한다 (검색 결과 스니펫만으로 건너뛰지 않는다)
  3. must_include의 모든 서비스가 최종 목록에 포함되었는지 검증한다.
     하나라도 빠지면 해당 서비스를 직접 검색하여 추가한다.
  4. 최종 목록이 7개 미만이면 "서비스 부족"을 반환한다.
     Service Scout 에이전트가 키워드를 바꿔서 이 스킬을 재호출한다.
  5. 뉴스 기사, 블로그 글, 리뷰 사이트 자체는 서비스 목록에 포함하지 않는다.
     실제 서비스의 공식 URL만 수집한다.
```

```markdown
## 스킬: search-services

### 입력
- domain: 서비스 도메인 (예: "마케팅 자동화")
- keywords: 검색 키워드 목록
- scope: "국내" | "해외" | "전체"
- must_include: 반드시 포함할 서비스명 목록

### 실행 절차 (반드시 이 순서를 따른다)

**Step 1: 직접 검색 (최소 2회)**
[WebSearch] "{domain} 서비스"
[WebSearch] "{domain} 플랫폼"
→ 결과에서 서비스 공식 URL 추출

**Step 2: 비교/리뷰 검색 (최소 2회)**
[WebSearch] "{domain} 서비스 비교"
[WebSearch] "best {keywords[0]} tools"
→ 비교 글 URL 발견 시:
  [mcp_web_fetch] 비교 글 본문 수집
  → 본문에서 서비스명 + URL 추출 (최소 3개 이상 추출 시도)

**Step 3: 특화 검색 (최소 1회)**
[WebSearch] 도메인 특화 키워드 조합 (keywords에서 선택)

**Step 4: scope별 추가 검색**
scope가 "국내" 또는 "전체"이면:
  [WebSearch] 한국어 키워드로 최소 1회 추가 검색
scope가 "해외" 또는 "전체"이면:
  [WebSearch] 영어 키워드로 최소 1회 추가 검색

**Step 5: must_include 검증**
must_include의 각 서비스가 현재 목록에 있는지 확인.
없는 서비스가 있으면:
  [WebSearch] "{서비스명} 공식 사이트"
  → URL 확보하여 목록에 추가

**Step 6: 필터링 및 검증**
- 뉴스 기사, 블로그, 리뷰 사이트 자체 URL을 제거한다.
- 중복 URL을 제거한다.
- 서비스가 7개 미만이면 status: "insufficient"를 반환한다.

### 출력
{
  "status": "sufficient" | "insufficient",
  "services": [
    {
      "name": "서비스명",
      "url": "https://...",
      "source": "직접 검색 | 비교 글 추출 | must_include 지정",
      "brief": "검색 스니펫에서 추출한 한 줄 설명"
    }
  ],
  "search_log": [
    "검색 수행 기록 (어떤 쿼리로 몇 개 발견했는지)"
  ]
}
```

### 스킬 2 — crawl-page

```
목적: 단일 URL의 페이지 콘텐츠를 수집하여 반환한다.
      mcp_web_fetch를 먼저 시도하고, 부족하면 browser로 전환한다.

소속: Service Scout (다른 에이전트/플로우에서도 재사용 가능)

사용 도구: mcp_web_fetch, cursor-ide-browser

가드레일:
  1. mcp_web_fetch를 먼저 시도한다. browser부터 시작하지 않는다.
  2. mcp_web_fetch 결과가 500자 미만이면 "콘텐츠 부족"으로 판단하고
     browser로 재수집한다.
  3. mcp_web_fetch가 에러를 반환하면 browser로 재수집한다.
  4. browser 사용 시 반드시 lock → snapshot → unlock 순서를 따른다.
  5. browser로도 수집 실패하면, 실패 사유를 명시하고 
     빈 콘텐츠로 반환한다. 추정으로 채우지 않는다.
  6. 수집된 콘텐츠가 50,000자를 초과하면 앞에서부터 50,000자까지만 포함한다.
```

```markdown
## 스킬: crawl-page

### 입력
- url: 수집할 페이지 URL
- page_type: "main" | "features" | "demo" | "use_cases"
  (수집 후 어떤 유형의 페이지인지 태깅용)

### 실행 절차

**Step 1: mcp_web_fetch 시도**
[mcp_web_fetch] url
→ 성공 + 콘텐츠 500자 이상? → Step 4로 이동
→ 성공 + 콘텐츠 500자 미만? → Step 2로 이동
→ 에러? → Step 2로 이동

**Step 2: cursor-ide-browser 시도**
[browser_navigate] url
[browser_lock]
[browser_snapshot]
[browser_unlock]
→ 스냅샷 텍스트가 있으면? → Step 4로 이동
→ 스냅샷도 실패? → Step 3으로 이동

**Step 3: 수집 실패 처리**
수집 실패로 판단. 실패 사유 기록.
빈 콘텐츠로 반환 (추정이나 대체 텍스트를 넣지 않는다).

**Step 4: 결과 정리**
- 콘텐츠가 50,000자 초과 시 50,000자에서 자른다.
- 수집 방법 기록 (mcp_web_fetch | cursor-ide-browser)

### 출력
{
  "url": "https://...",
  "page_type": "features",
  "status": "success" | "partial" | "failed",
  "method": "mcp_web_fetch" | "cursor-ide-browser",
  "fail_reason": null | "403 차단" | "콘텐츠 부족" | "타임아웃",
  "content": "수집된 마크다운/스냅샷 텍스트",
  "content_length": 12500,
  "internal_links": ["페이지 내에서 발견된 내부 링크 URL 목록"]
}
```

### 스킬 3 — scan-navigation

```
목적: 서비스 메인 페이지에서 GNB(Global Navigation Bar) 구조를 
      체계적으로 추출한다.

소속: Service Scout

사용 도구: cursor-ide-browser

가드레일:
  1. 반드시 browser를 사용한다 (mcp_web_fetch로는 드롭다운 메뉴를 
     펼칠 수 없다).
  2. 먼저 snapshot으로 기본 GNB를 파악한다.
  3. GNB에 드롭다운/서브메뉴가 있으면 각 메뉴를 click하여 펼치고
     다시 snapshot으로 하위 구조를 수집한다.
  4. 메뉴 깊이(depth)는 최대 3단계까지만 탐색한다.
     3단계 이상은 "depth 3+" 로 표기한다.
  5. 각 메뉴 항목의 연결 URL을 함께 수집한다.

분리 이유:
  일반 페이지 수집(crawl-page)과 다른 점은, 
  메뉴를 "클릭하여 펼쳐야" 하위 구조가 드러난다는 것이다.
  crawl-page는 페이지를 그대로 읽는 것이고,
```

### 스킬 4 — build-feature-matrix

```
목적: 서비스별 기능 분석 결과를 받아서 일관된 기준의 
      비교 매트릭스를 생성한다.

소속: Feature Analyzer

사용 도구: 없음

가드레일:
  1. 매트릭스의 평가 기준 4단계(●/○/△/✕)를 일관되게 적용한다.
  2. 첫 번째 서비스에 적용한 기준을 마지막 서비스에도 동일하게 적용한다.
  3. 평가 판단의 근거를 각 셀마다 한 줄로 기록한다.
     (근거 없이 기호만 적지 않는다)
  4. "수집 데이터에서 확인 안 됨"과 "해당 기능 없음"을 구분한다.
     확인 안 된 것은 ?(물음표)로 별도 표기한다.

분리 이유:
  에이전트 프롬프트에 매트릭스 작성 규칙을 넣어두면,
  서비스가 10개를 넘어가면서 앞쪽과 뒤쪽의 평가 기준이 달라지는 문제가 발생한다.
  스킬로 분리하여 "한 기능씩, 모든 서비스를 횡으로 비교"하는 절차를 강제한다.
```

```markdown
## 스킬: build-feature-matrix

### 입력
- services: 서비스 이름 목록
- features_by_service: 서비스별 기능 분석 결과 
  (Feature Analyzer 2단계까지의 카테고리별 기능 목록)

### 실행 절차

**Step 1: 통합 기능 목록 생성**
모든 서비스의 기능을 합쳐서 중복을 제거한 전체 기능 목록을 만든다.
기능 카테고리별로 그룹핑한다.

**Step 2: 기능별 횡단 비교 (가드레일 핵심)**
전체 기능 목록의 각 기능에 대해, 모든 서비스를 횡으로 비교한다.
(서비스 단위가 아니라 기능 단위로 순회한다)

각 셀의 평가 기준:
- ●: 수집 데이터에서 해당 기능이 명시적으로 설명되어 있고,
     상세 기능이나 활용 사례가 함께 언급됨
- ○: 수집 데이터에서 해당 기능이 언급되어 있지만,
     상세 설명 없이 기능 목록에만 포함됨
- △: 정확히 같은 기능은 아니지만, 유사한 기능이 다른 방식으로 존재함
     (반드시 어떤 점이 다른지 근거 기록)
- ✕: 수집 데이터 전체에서 해당 기능에 대한 언급을 찾을 수 없음
- ?: 해당 서비스의 관련 페이지 수집이 실패하여 판단 불가

**Step 3: 근거 기록**
매트릭스의 각 셀에 대해 판단 근거를 한 줄로 기록한다.
근거 예시:
- ● "기능 페이지에서 'Advanced Automation Builder'로 상세 설명"
- ○ "메인 페이지 기능 목록에 'Automation' 포함, 상세 없음"
- △ "자동화 대신 'Rules Engine'으로 유사 기능 제공"
- ✕ "기능 페이지, 메인 페이지 모두에서 언급 없음"
- ? "기능 페이지 수집 실패 (403)"

### 출력

**기능 비교 매트릭스**
| 카테고리 | 기능 | 서비스A | 서비스B | 서비스C | ... |
|---------|------|---------|---------|---------|-----|
| 핵심 | 기능1 | ● | ○ | ✕ | |
| 핵심 | 기능2 | ○ | ● | △ | |
| AI | 기능3 | ● | ? | ✕ | |

**평가 근거**
| 기능 | 서비스A | 서비스B | 서비스C |
|------|---------|---------|---------|
| 기능1 | 기능 페이지 상세 설명 | 메인 페이지 목록에만 | 언급 없음 |
| 기능2 | 목록에만 포함 | 데모 페이지에서 시연 | Rules Engine으로 유사 구현 |

**통계 요약**
- 전체 기능 수: N개
- 70% 이상 보유 (필수 후보): [기능 목록]
- 30% 미만 보유 (차별화 후보): [기능 목록]
- ? 표기 (데이터 부족): [서비스 × 기능 목록]
```

### 스킬 5 — build-uxui-comparison

```
목적: 서비스별 UXUI 분석 결과를 받아서 일관된 기준의 
      비교표를 생성한다.

소속: UXUI Analyzer

사용 도구: 없음

가드레일:
  1. 비교 항목을 표준화하여 모든 서비스에 동일하게 적용한다.
  2. 정량적으로 비교 가능한 항목(GNB 항목 수, 메뉴 깊이, CTA 수)과
     정성적 항목(디자인 철학, 톤앤매너)을 구분하여 표를 분리한다.
  3. 정성적 비교는 미리 정의된 유형 분류를 사용한다.
     자유 서술이 아니라 유형 + 한 줄 근거 형태로 통일한다.
  4. "수집 데이터에서 확인 안 됨"은 별도 표기한다.

분리 이유:
  Feature Analyzer의 매트릭스와 동일한 이유.
  서비스가 많아지면 비교 기준이 흔들린다.
  특히 UXUI는 정성적 판단이 많아서 일관성 확보가 더 어렵다.
  비교 유형을 미리 정의하고 그 안에서 선택하게 하는 것이 핵심이다.
```

```markdown
## 스킬: build-uxui-comparison

### 입력
- services: 서비스 이름 목록
- uxui_by_service: 서비스별 UXUI 분석 결과
  (UXUI Analyzer의 서비스별 분석)

### 실행 절차

**Step 1: 정량 비교표 생성**

| 서비스 | GNB 항목 수 | 메뉴 깊이 | CTA 유형 | 메인 콘텐츠 섹션 수 |
|--------|-----------|----------|---------|-------------------|
| A | 5 | 2 | 무료 체험 | 6 |
| B | 7 | 3 | 데모 요청 | 8 |

**Step 2: 정성 비교표 생성 (유형 분류 기반)**

각 비교 항목에 대해 미리 정의된 유형 중에서 선택하고, 한 줄 근거를 기록한다.

비교 항목과 유형 정의:

**정보 구조 유형**
- 기능 중심: GNB가 기능/제품별로 분류 (예: "Products > Email, Automation, CRM")
- 솔루션 중심: GNB가 사용자/목적별로 분류 (예: "For Marketers, For Sales")
- 하이브리드: 기능 + 솔루션 혼합

**CTA 전략 유형**
- 무료 체험 유도: "Start Free", "무료 시작"
- 데모 요청 유도: "Request Demo", "데모 신청"
- 가입 유도: "Sign Up", "회원가입"
- 콘텐츠 유도: "Learn More", "자세히 보기"

**콘텐츠 전략 유형**
- 가치 중심: 기능보다 결과/효과를 강조 ("매출 30% 증가")
- 기능 중심: 기능의 상세 스펙을 강조 ("드래그 앤 드롭 에디터")
- 사례 중심: 고객 사례와 수치를 중심으로 설득

**톤앤매너 유형**
- 전문적/엔터프라이즈: 격식체, 전문 용어, 무거운 톤
- 친근한/스타트업: 구어체, 이모지, 가벼운 톤
- 미니멀/심플: 짧은 문장, 여백 강조, 핵심만
- 정보 밀집: 많은 텍스트, 상세한 설명

**복잡도 처리 유형**
- 단계별 가이드: 1→2→3 스텝 형태로 안내
- 탭/아코디언: 정보를 숨기고 필요 시 펼침
- 시각화: 도표/다이어그램으로 복잡도 해소
- 시나리오 기반: 사용 사례별로 기능을 묶어서 제시

정성 비교표:
| 서비스 | 정보 구조 | CTA 전략 | 콘텐츠 전략 | 톤앤매너 | 복잡도 처리 |
|--------|----------|---------|-----------|---------|-----------|
| A | 기능 중심 | 무료 체험 | 가치 중심 | 친근한 | 단계별 가이드 |
| B | 솔루션 중심 | 데모 요청 | 사례 중심 | 전문적 | 시나리오 기반 |

각 셀의 근거:
| 서비스 | 정보 구조 근거 | CTA 전략 근거 | ... |
|--------|--------------|-------------|-----|
| A | GNB: Products/Pricing/Resources | Hero 영역 "Start Free" 버튼 | |
| B | GNB: Solutions(by role)/Platform | Hero 영역 "Get a Demo" 버튼 | |

**Step 3: 패턴 도출**
비교표에서:
- 가장 많은 서비스가 선택한 유형 = 업계 공통 패턴
- 소수만 선택한 유형 = 차별화 패턴
- 특정 유형 조합이 반복되면 = 트렌드

### 출력
- 정량 비교표
- 정성 비교표 (유형 + 근거)
- 업계 공통 패턴 목록
- 차별화 패턴 목록
```
