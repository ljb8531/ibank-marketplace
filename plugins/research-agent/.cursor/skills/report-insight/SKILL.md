# 리포트 인사이트·실행방안·최종정리 생성 (항목 4, 10, 11, 12)

## 입력

$ARGUMENTS = 앞선 3개 섹션 파일 경로 또는 요약. report-writer가 전달.

## 경로 (OUTPUT_BASE)

프로젝트 루트의 `.benchmark-output-root`를 읽어 OUTPUT_BASE 확정(없으면 `output`). 모든 경로는 `{OUTPUT_BASE}/output/...`.

## 참조 데이터

- {OUTPUT_BASE}/output/reports/sections/01-asis.md, 02-deepdive.md, 03-matrix.md
- {OUTPUT_BASE}/output/data/reference-codes.json
- {OUTPUT_BASE}/output/data/project-brief.json
- {OUTPUT_BASE}/output/data/raw-input.txt (고객 요구사항·요청 방향 반영 시)

## 원칙

- "그래서 우리에게 가져다주는 인사이트는 무엇인가"를 **구체적으로** 도출한다. 포괄적·두루뭉술한 표현은 사용하지 않는다.
- 모든 인사이트·실행방안에 참조 코드(PP-n, D-n, I-n, R-n)를 포함해 앞선 분석과 논리적으로 연결한다.

## 실행 절차

### Part A: 4. 특화·선도 트렌드

**"4. 특화·선도 트렌드"** 제목. 요청 내용 이외에, 벤치마킹·리서치 과정에서 발견한 **톡톡 튀는 아이디어·업계를 선도하는 트렌드**를 정리한다.

- **입력**: 01-asis(목적·범위·시장 트렌드), 02-deepdive(사이트별 강점·차별 요소), 03-matrix(갭·공통 트렌드).
- **내용**: (1) 분석 요청에 명시되지 않았으나 벤치마킹 대상에서 공통으로 관찰된 선도적 패턴 1~3가지. (2) Indirect/Inspire에서 발견한 타 업계 선도 사례·차용 가능 아이디어. (3) "업계를 선도하는" 수준의 트렌드일 때 구체적 사이트명·기능/UX 예시를 함께 기재.
- 산문 또는 불릿. 추상적 나열 금지 — 어떤 사이트에서 어떤 방식으로 나타나는지 구체적으로.

### Part B: 10. 핵심 인사이트

**"10. 핵심 인사이트"** 제목. 01~03 섹션을 Read한 뒤 **"따라서 우리에게 가져다주는 인사이트는 무엇인가"**를 3가지 카테고리로 구체적으로 서술. 포괄적·두루뭉술한 문장은 사용하지 않는다.

**10-1. Table Stakes — 반드시 갖춰야 할 요소**

경쟁사 과반(Direct 3개 이상)이 이미 채택한 기능/UX/디자인. 없으면 시장 기본 기대치 미충족.

- **TS-1, TS-2, TS-3 …**: 요소명 + 구체적 서술. 어떤 D-n에서 관찰했는지, 어떤 PP-n을 해소하는지, 구현 시 참고 벤치마크(I-n 또는 D-n). "~가 필요하다" 수준이 아닌, "D-1의 [구체적 기능/화면]처럼 [무엇을] 도입하면 [PP-n]을 [이렇게] 해소할 수 있다" 형태.

**10-2. White Space — 차별화 기회**

경쟁사 대부분이 미비한 영역. 주로 Indirect(I-n)에서 발견한 크로스 인더스트리 인사이트.

- **WS-1, WS-2 …**: 기회명 + 구체적 서술. 03-matrix 갭 분석에서 경쟁사가 ○인 영역에 I-n 접근을 적용하면 차별화 가능한 점. "[I-n]의 [기능/UX]를 우리 서비스에서 [이렇게] 변형하면 [PP-n] 해결과 차별화 동시에 가능."

**10-3. Leapfrog vs. Avoid — 전략적 판단 필요**

특정 D-n이 압도적으로 앞선 영역. 정면 경쟁 대신 우회·뛰어넘기 필요.

- **LA-1 …**: 영역명 + 해당 영역·압도적 경쟁사(D-n)·정면 경쟁 회피 이유·대안적 접근(우회/뛰어넘기) 제안.

### Part C: 11. 실행 방안/전략

**"11. 실행 방안/전략"** 제목. 고객사 요구사항·트렌드 조사·벤치마킹 인사이트를 기반으로 **구축/개발 방향·컨셉과 구현 전략**을 세우고, **현실적·구체적인 실행 방안**을 제시한다. 다음 내용이 반드시 포함되어야 한다.

**11-1. 구축 방향·컨셉**  
인사이트 종합. 리뉴얼/신규의 포지셔닝·핵심 가치 제안. core_purpose 대비 강화/조정. Table Stakes로 기본 확보, White Space로 차별화, Leapfrog/Avoid로 리소스 배분 전략.

**11-2. UI/UX 화면구성·레이아웃·화면기획**

- **메인페이지**: 컨셉, 필수 블록(히어로·카테고리·CTA·트러스트 등), 레이아웃 원칙(그리드·폴드·스크롤), 벤치마크 참조(D-n, R-n).
- **서브페이지**: 주요 서브 유형별(예: 상세·목록·마이페이지) 화면구성 원칙, 필수 요소, IA·네비게이션 방향. 02-deepdive·03-matrix에서 도출한 패턴을 "우리 서비스에 적용 시"로 서술.

**11-3. 필수 요소·기능적 아이디어**

| 구분 | 항목 | 설명 | 참조(PP-n, TS-n, D-n/I-n/R-n) |
|------|------|------|-------------------------------|
| 필수 요소 | … | 메인/서브에 반드시 넣을 UI·기능 요소 | |
| 기능적 아이디어 | … | 벤치마킹에서 차용·변형할 기능·인터랙션 | |

각 행에 구체적 명칭·동작·기대 효과를 1~2문장으로 기재. "~를 검토한다" 수준이 아닌, "~를 [이렇게] 구현하면 [PP-n/TS-n]에 기여한다" 형태.

**11-4. 핵심 기능 우선순위 테이블**

| 우선순위 | 기능 | 근거 | 참조 | 연결 인사이트 |
|---------|------|------|------|---------------|
| Must-Have | … | PP-n 해소 + TS-n | D-n, I-n | Table Stakes |
| Nice-to-Have | … | WS-n 차별화 | I-n | White Space |
| Future | … | 기술 성숙도 또는 LA-n | D-n | Leapfrog |

**11-5. 디자인 방향 권고**  
03-matrix 디자인 비교 결과 기반. 정보 밀도·비주얼 포커스·컬러·타이포·레이아웃. "R-n의 [요소] + R-n의 [요소] 참고, [자사 특성]으로 [조정]" 형태.

**11-6. 실행 로드맵 테이블**

| Phase | 기간 | 핵심 작업 | 연결 인사이트 | 핵심 산출물 |
|-------|------|----------|---------------|------------|
| Phase 1: Quick Win | 1~2개월 | PP-n 중 기술 난이도 낮은 즉시 개선 | TS-n | 구체적 산출물 |
| Phase 2: Core Build | 3~6개월 | Must-Have 구현, 디자인 시스템·메인/서브 기획안 | TS-n, WS-n | 구체적 산출물 |
| Phase 3: Optimization | 6개월~ | Nice-to-Have/Future, 데이터 기반 개선 | WS-n, LA-n | 구체적 산출물 |

### Part D: 12. 최종정리

**"12. 최종정리"** 제목. 전체 리포트를 한 페이지 분량으로 압축 정리한다.

- **핵심 결론**: 이번 분석이 도달한 최종 방향 1~2문장. "[회사/서비스]는 [방향]을 중심으로 [리뉴얼/구축]을 추진하고, [Table Stakes 요약]을 확보한 뒤 [White Space 1가지]로 차별화하는 것이 유리하다" 수준.
- **우선 실행 3가지**: 의사결정자가 당장 착수할 수 있는 액션 3개. PP-n 해소·TS-n 반영·Phase 1 Quick Win과 연결.
- **참고**: 상세는 본문 1~11항목 및 부록 참조.
